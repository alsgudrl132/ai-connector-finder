# AI Connector Finder — STM32 ↔ Pi UART 프로토콜 스펙

## 1. 프레임 구조

```
AA    CMD    LEN    [PAYLOAD...]    CHK    55
```

| 필드 | 크기 | 의미 |
|---|---|---|
| **START** | 1B | 프레임 시작을 알리는 고정값. 항상 `AA`. UART 스트림 중간에서 파서가 "여기서부터 새 프레임 시작"임을 인식하는 동기화 기준점 |
| **CMD** | 1B | 명령/이벤트 코드. 어떤 동작을 요청하는지, 또는 어떤 응답/이벤트인지를 나타냄 |
| **LEN** | 1B | 뒤따라올 PAYLOAD가 몇 바이트인지 미리 알려주는 길이값. 파서가 PAYLOAD를 몇 바이트 읽고 멈출지 결정하는 기준. `00`이면 PAYLOAD 없음 |
| **PAYLOAD** | 0~N B | 명령의 실제 데이터(파라미터). LEN에 적힌 만큼의 바이트 수만 존재 |
| **CHK** | 1B | 체크섬. `START ^ CMD ^ LEN ^ PAYLOAD(전부)`의 XOR 값. 통신 중 깨진 바이트를 검출하는 용도 |
| **END** | 1B | 프레임 끝을 알리는 고정값. 항상 `55` |

**왜 LEN이 PAYLOAD보다 먼저 오는가**: 파서는 바이트를 순서대로 하나씩 읽는다. PAYLOAD를 몇 바이트 읽어야 할지 미리 알아야 정확히 그만큼만 읽고 다음 필드(CHK)로 넘어갈 수 있다. 그래서 "길이를 먼저 예고 → 그다음 실제 값" 순서(TLV: Type-Length-Value 패턴)가 강제된다.

## 2. CMD 코드 표

### Pi → STM32 (하드웨어 제어 요청)

| CMD 값 | 이름 | PAYLOAD | 의미 |
|---|---|---|---|
| `01` | HOME | 없음 | 캐러셀 원점 복귀 (리밋스위치까지) |
| `02` | STEP | 없음 | 캐러셀 한 칸 회전 |
| `03` | OPEN | 없음 | 슬라이드 덮개 열기 |
| `04` | CLOSE | 없음 | 슬라이드 덮개 닫기 |
| `05` | LED_SET | 1B (색상코드) | RGB LED 상태 변경 |
| `06` | LCD_MSG | N바이트 (문자열) | LCD1602 출력 |
| `07` | ABORT | 없음 | 현재 동작 즉시 중단 |

### STM32 → Pi (요청/응답/이벤트)

| CMD 값 | 이름 | PAYLOAD | 의미 |
|---|---|---|---|
| `10` | SCAN_REQ | 없음 | 현재 칸 카메라 판독 요청 |
| `11` | BUTTON_EVENT | 1B (버튼ID) | 물리 버튼 눌림 알림 |
| `81` | ACK_OK | 없음 | 명령 정상 처리 완료 (범용) |
| `82` | ACK_OK_SLOT | 1B (슬롯번호) | STEP 완료, 현재 슬롯 번호 응답 |
| `91` | SCAN_RESULT | 1B (0x00/0x01) | Pi 판독 결과 (불일치/일치) |
| `FF` | ERR | 1B (에러코드) | 에러 발생 |

## 3. 바이트 하나하나 뜯어보기 (예제)

### 예제 A — `AA 10 00 BA 55` (SCAN_REQ)

| 바이트 | 필드 | 해석 |
|---|---|---|
| `AA` | START | 프레임 시작 |
| `10` | CMD | SCAN_REQ (현재 칸 판독해달라는 요청) |
| `00` | LEN | PAYLOAD 0바이트 → 이 명령은 추가 데이터 없이 그 자체로 의미가 끝남 |
| — | PAYLOAD | 없음 (LEN=0이므로 이 필드 자체가 존재하지 않음) |
| `BA` | CHK | `AA^10^00 = BA` |
| `55` | END | 프레임 끝 |

### 예제 B — `AA 91 01 01 3B 55` (SCAN_RESULT, 일치)

| 바이트 | 필드 | 해석 |
|---|---|---|
| `AA` | START | 프레임 시작 |
| `91` | CMD | SCAN_RESULT (판독 결과 응답) |
| `01` | LEN | PAYLOAD 1바이트가 뒤에 온다는 예고 |
| `01` | PAYLOAD | 실제 값. `01` = 일치(찾는 커넥터 맞음). 참고로 `00`이면 불일치 |
| `3B` | CHK | `AA^91^01^01 = 3B` |
| `55` | END | 프레임 끝 |

### 예제 C — `AA FF 01 02 56 55` (ERR, 서보 응답없음)

| 바이트 | 필드 | 해석 |
|---|---|---|
| `AA` | START | 프레임 시작 |
| `FF` | CMD | ERR (에러 알림) |
| `01` | LEN | PAYLOAD 1바이트 예고 |
| `02` | PAYLOAD | 에러코드 `02` = 서보 응답없음 (에러코드 표는 4번 항목 참고) |
| `56` | CHK | `AA^FF^01^02 = 56` |
| `55` | END | 프레임 끝 |

## 4. 서브 코드 표 (PAYLOAD 값 상세)

### LED_SET 색상코드
| 값 | 의미 |
|---|---|
| `01` | 초록 (매칭 성공) |
| `02` | 빨강 (에러) |
| `03` | 노랑 (스캔중/대기) |

### BUTTON_EVENT 버튼ID
| 값 | 의미 |
|---|---|
| `01` | SCAN 버튼 |
| `02` | OK 버튼 |
| `03` | CANCEL 버튼 |

### SCAN_RESULT 값
| 값 | 의미 |
|---|---|
| `00` | 불일치 |
| `01` | 일치 |

### ERR 에러코드
| 값 | 의미 |
|---|---|
| `01` | 리밋스위치 타임아웃 (HOME 실패) |
| `02` | 서보 응답없음 |
| `03` | 알 수 없는 CMD |
| `04` | 체크섬 불일치 |

## 5. 체크섬 계산 규칙

```
CHK = START ^ CMD ^ LEN ^ PAYLOAD[0] ^ PAYLOAD[1] ^ ... ^ PAYLOAD[n-1]
```

XOR 누적 계산. PAYLOAD가 없으면 `START ^ CMD ^ LEN`까지만 계산.

**검증 시 주의**: 이 문서의 CHK 값은 참고용 계산 예시이므로, 실제 구현한 파서/송신 코드에서 나온 값과 반드시 직접 대조해서 확인할 것.

## 6. LCD_MSG — "찾는 커넥터 이름"은 별도 CMD가 필요 없는 이유

**질문**: "USB-A를 찾고 있다"는 정보를 STM32한테 알려주려면 새로운 CMD가 필요한가?

**결론: 필요 없음.** 이미 있는 `06 LCD_MSG`를 그대로 쓰면 된다. 이유는 STM32가 그 값을 "이해"할 필요가 없기 때문이다.

- **STM32가 값을 알아야(=로직 분기에 써야) 하는 경우** → 새 CMD/의미 정의가 필요함 (예: "USB-A면 A동작, XT60이면 B동작"처럼 STM32 내부 로직이 값에 따라 갈라질 때)
- **STM32가 값을 그냥 표시만 하면 되는 경우** → 어떤 문자열이든 그대로 LCD에 찍기만 하면 되므로, 기존 LCD_MSG 프레임에 그 내용을 PAYLOAD로 실어 보내면 끝

목표 커넥터가 "USB-A"라는 사실은 **AI판정 단계에서 Pi가 스스로 알아낸 정보**이고, STM32 쪽 로직(캐러셀 돌리기, SCAN_REQ 보내기, SCAN_RESULT 받기)은 이 값과 무관하게 동일하게 동작한다. 그래서 STM32는 "USB-A"라는 값의 의미를 몰라도 되고, 그저 LCD에 찍을 바이트 뭉치로만 취급하면 된다. (비유: 프린터는 인쇄할 문서 내용을 이해할 필요 없이 그대로 찍어내기만 하면 됨)

**중요**: "프로토콜이 필요 없다"가 아니라 **"새 CMD를 안 만들어도 된다"**는 뜻. 여전히 기존 프레임 구조(START/CMD/LEN/PAYLOAD/CHK/END) 그대로 사용하고, PAYLOAD 자리에 문자열의 아스키 바이트가 들어갈 뿐이다.

### 예제 — Pi가 AI판정 직후 목표를 LCD에 표시 (`AA 06 05 55 53 42 2D 41 <chk> 55`)

| 바이트 | 필드 | 해석 |
|---|---|---|
| `AA` | START | 프레임 시작 |
| `06` | CMD | LCD_MSG |
| `05` | LEN | PAYLOAD 5바이트 ("USB-A"가 5글자) |
| `55 53 42 2D 41` | PAYLOAD | "U", "S", "B", "-", "A"의 아스키 코드값. STM32는 이 값의 의미(커넥터 종류)를 몰라도 되고, 그대로 `lcd_print(payload)`처럼 화면에 출력만 하면 됨 |
| `<chk>` | CHK | `AA^06^05^55^53^42^2D^41` |
| `55` | END | 프레임 끝 |

이 흐름은 AI판정 완료 직후 한 번만 보내면 되고, 이후 순차인식(STEP/SCAN_REQ/SCAN_RESULT) 반복 구간에서는 목표 이름이 다시 오갈 필요가 없다.

## 7. 설계상 미결정 사항 (구현 전 결정 필요)

1. **통신 방향 원칙 충돌**: SCAN_REQ(`10`), BUTTON_EVENT(`11`)는 STM32가 먼저 프레임을 보내는 케이스. "Pi=마스터, 항상 먼저 요청" 원칙과 상충하므로, 폴링 방식으로 바꿀지 양방향 허용하고 파서에서 구분할지 결정 필요.
2. **LCD_MSG 가변 길이**: 문자열 통째로 보내는 대신 "상태 코드 1바이트 + STM32 내부에 문구 미리 저장" 방식이 파싱 단순화 측면에서 더 나을 수 있음.
3. **타임아웃 값**: Pi가 응답을 얼마나 기다릴지 (HOME/STEP/OPEN/CLOSE 각각 동작 시간이 다르므로 커맨드별 타임아웃 분리 고려).
